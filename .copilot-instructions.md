## Build, run, test, and migration commands

Backend (FastAPI, Python)
- Install deps: python -m venv venv && venv\Scripts\activate && pip install -r Backend\requirements.txt
- Run migrations: alembic upgrade head (alembic.ini configured to a local Postgres URL)
- Seed DB (dev): python Backend\seeds.py
- Wipe DB (dev): python Backend\wipe_db_final.py
- Run server (local dev): uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
- Notes: configuration uses pydantic-settings and an `.env` file at Backend/.env (see Backend/app/core/config.py)

Frontend (Angular)
- Install: npm install (in Frontend/)
- Dev server: npm run start (or ng serve)
- Build: npm run build
- Tests (Vitest via Angular CLI): npm run test  (to run the full suite)
- Run a single test (example): npx vitest -t "pattern"  or use Vitest CLI to target specific tests

Mobile (Flutter)
- Install deps: flutter pub get (in mobile/)
- Run on device/emulator: flutter run -d <device>
- Tests: flutter test
- Run a single test (example): flutter test test/path/to_test.dart -n "test name pattern"

## High-level architecture (big picture)

- Backend/ (FastAPI)
  - app/ contains domain modules under app/modules/* (usuarios, vehiculos, emergencias).
  - app/main.py composes FastAPI app, registers routers and CORS.
  - app/core holds shared infra: database engine/session, config (pydantic), and security helpers.
  - Persistence uses SQLAlchemy with Alembic migrations; alembic.ini points to a local PostgreSQL DB by default.
  - Media uses Cloudinary; secrets are loaded from Backend/.env via pydantic-settings.

- Frontend/ (Angular)
  - Standard Angular CLI app (package.json scripts: start, build, test).
  - Tailwind + PostCSS and Vitest for unit testing are configured as devDependencies.

- mobile/ (Flutter)
  - Flutter application using plugins: geolocator, permission_handler, flutter_dotenv, flutter_secure_storage, web_socket_channel, etc.
  - pubspec.yaml includes .env as an asset (app reads runtime env via flutter_dotenv).

Integration: Clients (Angular web and Flutter mobile) communicate with FastAPI backend (REST endpoints). DB is Postgres. Static/media stored via Cloudinary.

## Key conventions and repo-specific patterns

- Backend module layout: each domain under app/modules/<name> contains models, routes, and services. Routers are imported and included in app/main.py.
- Config: use app.core.config.settings for secrets and toggles (loads Backend/.env). Avoid direct os.getenv in new code; prefer settings.
- DB sessions: use SessionLocal from app.core.database in one-off scripts (seeds, wipe) and dependency injection for request-scoped sessions in routes.
- Alembic: migrations live under Backend/alembic. Use alembic commands from repo root but ensure DB URL/env is correct.
- Seeds and wipe scripts live in Backend/ and can be executed directly for test data or teardown.
- Frontend tests: run via `npm run test` (Vitest). To run an individual test, call Vitest directly (npx vitest -t "name-pattern").
- Mobile env: .env file is packaged as an asset; secrets should be injected carefully (don't commit sensitive values).
